# Audit: PCG Mapping UI (F4)

Here is a technical audit of the current implementation for the PCG Mapping interface based on the codebase.

### 1. File Location
The mapping UI is implemented in:
**`src/pages/Mapping.tsx`**

*(Note on layout layout terminology: The interface actually renders the Target PCG Tree in the Left Pane and the Unmapped Accounts in the Right Pane, which is standard for left-to-right mapping workflows).*

### 2. Data Structures
The component uses two main TypeScript interfaces for state management:

**Unmapped Source Accounts (Right Pane):**
```typescript
export interface UnmappedAccount {
    accountId: string;
    accountName: string;
    balance: number;
}
```

**Target PCG Structure (Left Pane):**
```typescript
export interface TreeNode {
    id: string;
    type: 'category' | 'pals_account' | 'mapped_account';
    label: string;
    children?: TreeNode[];
    isOpen?: boolean;
    // Only present if type === 'mapped_account'
    sourceAccountId?: string;
    balance?: number;
}
```

### 3. Drag and Drop Mutator (`handleDragEnd`)
When an account is dragged from the unmapped list and dropped onto the target tree, the `@dnd-kit/core` `onDragEnd` event fires `handleDragEnd`. 

**What it mutates:**
1. It removes the dragged item from the `unmappedAccounts` array state.
2. It recursively finds the target node in the `treeData` state and pushes a new `TreeNode` (as a child) to visually represent the mapping.

```typescript
const handleDragEnd = (event: DragEndEvent) => {
    const { active, over } = event;
    setActiveAccount(null);

    // If dropped outside a valid droppable area
    if (!over) return;

    const accountId = active.id as string;
    const targetNodeId = (over.id as string).replace('node-', '');
    const account = active.data.current?.account as UnmappedAccount;

    if (!account) return;

    // 1. Remove from Right Pane (State Mutation)
    setUnmappedAccounts(prev => prev.filter(a => a.accountId !== accountId));

    // 2. Add to Left Pane Tree (State Mutation)
    const addToTreeRecursive = (nodes: TreeNode[]): TreeNode[] => {
        return nodes.map(node => {
            if (node.id === targetNodeId) {
                const newMappedNode: TreeNode = {
                    id: `mapped-${account.accountId}`,
                    type: 'mapped_account',
                    label: `${account.accountId} - ${account.accountName}`,
                    sourceAccountId: account.accountId,
                    balance: account.balance
                };
                return {
                    ...node,
                    children: [...(node.children || []), newMappedNode],
                    isOpen: true // Auto-expand drop target
                };
            }
            if (node.children) {
                return { ...node, children: addToTreeRecursive(node.children) };
            }
            return node;
        });
    };

    setTreeData(addToTreeRecursive(treeData));
};
```

### 5. Optimistic Updates
**Yes.** The mapping is handled entirely via **local Optimistic Updates**. The drag-and-drop interactions rely purely on React State (`treeData` and `unmappedAccounts`). The user creates their final state visually, and the API is only intended to be called as a single batch operation when clicking "Sync with Engine". 

### 6. Error Handling
*   **Drag-and-Drop Errors:** If an item is dropped in an invalid zone, the `handleDragEnd` simply returns early, and the item "snaps" back to its original position via `@dnd-kit`'s default behavior.
*   **API Errors:** Because the API call is currently a `setTimeout` mock, there is no real error boundary, Toast notification, or rollback mechanism wired up for a failed save yet.
